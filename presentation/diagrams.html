Six views of the same system, simplest → most detailed. Pick one for the end-to-end slide.

Diagram 1 · simplest

At a glance

The whole product in one line: a pull request goes in, a deterministic score comes out, the dashboard reads it.

Diagram 2 · event flow

From a PR event to a dashboard row

A PR event fires a Harness trigger → a pipeline calls /analyze → the score is persisted and surfaced, and the result is written back onto the PR.

Diagram 3 · layered system

Sources → deterministic core → surfaces (with the AI on the side)

Both SCMs normalize through one adapter; the pure core scores; Claude reads the score (never changes it); FastAPI + Postgres serve the dashboard, write-back and Slack.

Diagram 4 · end-to-end sequence

Every step, PR open → dashboard

The exact order of operations. Steps 1–7 happen synchronously; 8–11 run in the background so the response never blocks.

Diagram 5 · hexagonal architecture

Ports & adapters — the core is pure

The domain + analyzers + scoring engine import nothing external. Every integration is an adapter on the edge, and all dependencies point inward — which is why one product covers two SCMs and two LLM backends.

Diagram 6 · scoring internals

How one PR's score is computed

Deterministic, end to end. The LLM is nowhere in this path. The Jira signal is surfaced but carries zero weight, so it never moves a number.
